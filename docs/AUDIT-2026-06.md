## 1. PATTERN DEBT

Every block editor now carries **two render paths in one file**: a slide-out panel
path (`panelMode`) and the legacy `Dialog` wrapper (fallback). All 13 editors have
`panelMode`, so the panel migration is functionally complete — but the Dialog-era
scaffolding still ships alongside it.

- **P2 · all 13 editors (e.g. `ContentSectionEditor.tsx:594`, `GalleryEditor.tsx:291`, `LinksEditor.tsx:237`) · The `-mx-6 px-6` negative-margin ScrollArea is now guarded by `panelMode ? 'flex-1 px-4' : 'flex-1 -mx-6 px-6'`, but the dead Dialog branch keeps the negative margin alive.** · Once the Dialog fallback is confirmed unreachable (all editors are opened as panels via `ProfileDashboard`), drop the ternary and the Dialog wrapper entirely.
- **P1 · `BioEditor.tsx:281`, `TextBlockEditor.tsx:289` · These two hardcode `bg-[#1a1a1a] border-white/10 text-white` on `DialogContent` instead of the panel/token system used by the other 11, and have no ScrollArea.** · They diverge visually from the slide-out panel chrome. Align to the shared panel surface tokens.
- **P1 · ~8 editors repeat `DialogContent className="sm:max-w-lg max-h-[90vh] flex flex-col"` verbatim (`FeaturedMediaEditor.tsx:556`, `GalleryEditor.tsx:372`, `LinksEditor.tsx:884`, `ProductCardsEditor.tsx:790`, `SocialIconRowEditor.tsx:639`, `SocialLinksEditor.tsx:723`, `ContentSectionEditor.tsx:656`, `EmailSubscribeEditor.tsx:303`).** · Hoist into one `<EditorShell panelMode>` wrapper; editors render only their body.
- **P1 · JSON-in-`title` style config parsed ad-hoc in ~20 sites with inconsistent fallbacks (`'{}'` vs `''` vs none): `LinksBlock.tsx:154`, `ContentSectionBlock.tsx:40`, `VideoFeedBlock.tsx:32`, `ProductCardsBlock.tsx:34`, `PrimaryCtaBlock.tsx:22`, `SocialIconRowBlock.tsx:39`, `EditableProfileView.tsx:208,242`, plus every editor's load path.** · Extract `parseBlockStyle(title): BlockStyleConfig` with one validated fallback; replaces the scattered `try { JSON.parse } catch {}`.
- **P2 · Three drag-preview style strings are copy-pasted across editors: `'opacity-90 shadow-lg scale-[1.01] border-primary/50 z-10'` (`ContentSectionEditor.tsx:124`, `FeaturedMediaEditor.tsx:106`, `ProductCardsEditor.tsx:132`, `SocialIconRowEditor.tsx:125`, `SocialLinksEditor.tsx:175`).** · Move to a shared `sortableDragClass` constant.

## 3. MOBILE RISK SWEEP (iOS)

- **P2 · `h-full` inside `aspect-ratio` parent.** GalleryEditor thumbnail was fixed (`absolute inset-0 w-full h-full`, `GalleryEditor.tsx:320`). Re-audit `GalleryBlock`/`ProductCardsBlock` public renders for any remaining `w-full h-full object-cover` inside an `aspect-square` without `absolute` — iOS collapses these to natural size. · Apply the same `absolute inset-0` fix wherever found.
- **P2 · `100vh` (not `100dvh`): `Editor.tsx:487` `h-[calc(100vh-52px)]`, `DeviceFrame.tsx:54,113` `calc(100vh-200px)`, `WelcomeCoach.tsx:178` `calc(100vh-80px)`; plus pervasive `min-h-screen`.** · iOS Safari's dynamic URL bar makes `100vh` overshoot the visible area. Switch the `calc()` cases to `100dvh`; consider a `min-h-[100dvh]` utility for full-height pages.
- **P2 · `fixed` inside transformed ancestors.** `DeviceFrame`/`LivePreviewPanel` apply `scale()` transforms to the phone frame; a `position: fixed` descendant would anchor to the transformed parent, not the viewport. · The gallery lightbox correctly portals to `body` (`EditableProfileView.tsx:1768`) — keep that pattern for anything `fixed` rendered from inside the preview.
- **P2 (low) · Inputs <16px.** The base `ui/input.tsx:11` uses `text-base` (16px) on mobile and only `md:text-sm` on desktop — **iOS-safe**. Risk only if an editor overrides an input with explicit `text-sm`/`text-xs`. · No input-level overrides found in the sweep; keep the base and avoid per-input shrinking.
- **P2 · Touch targets <44px.** `LinksBlock.tsx:80,94` drag/delete are `h-6 w-6` (24px); `EditableProfileView.tsx:415,474` carousel arrows `w-7 h-7` (28px), `:449` delete `w-8 h-8` (32px); `:283` lightbox close `w-10 h-10` (40px). · Wrap in a ≥44px hit area (padding or `before:` expander) without changing the visual glyph size.

**Count: 5**

## 5. TYPE SAFETY

- **P2 · `theme_json as any` — 12 sites** (`EditableProfileView.tsx:965,981,1004,1007,1010,1246,1533,1551,1557`; `PublicProfile.tsx:254`). The DB `theme_json` (`Json`) is cast to `any` then deep-accessed (`?.headerConfig?.nameCardY`, `?.avatar_url_page2`, `?.pages`). · Define a `ThemeJson` interface and cast once at the boundary; removes ~12 unsafe accesses and documents the shape.
- **P2 · `block.type as any` (`Editor.tsx:142`), `style_json as Record<string, any>` (`LinksBlock.tsx:210`, `LinksEditor.tsx:687`), `theme as any` (`DesignEditor.tsx:69`), `buttons/typography as Record<string, any>` (`LinkButton.tsx:116,117`).** · Narrow with the generated Supabase enums / a typed `StyleJson` shape.
- **P2 · `JSON.parse` shape is unvalidated (parse itself is try-wrapped, so runtime-safe).** `EditableProfileView.tsx:242` reads `parsed.layout`/`parsed.speed` with no guard; `HeroCardBlock.tsx:42` / `EmailSubscribeBlock.tsx:50` parse `item.badge` as JSON. · Fold into the `parseBlockStyle()` util (§1) with a validated return type, so callers get typed fields instead of `any`.
- **P2 · `(window as any).EyeDropper` (`DesignEditor.tsx:1299,1339`).** · Add a minimal `EyeDropper` ambient type rather than `any`.
- **Inventory only (protected) · `canvas as any` (`EditableProfileView.tsx:1304`), `(enhErr as any).context` (`:1412`)** — inside the hero/crop/enhance system. Flagged, not proposed for change.
- **Good · No widespread non-null assertions on Supabase rows** (`data!`) were found; error paths use `error: any` in catch blocks (acceptable, though `unknown` is preferable).

## 6. EDITOR / PREVIEW / PUBLIC PARITY

- **Good (low drift) · `PublicProfile.tsx:277` renders blocks by delegating to `EditableProfileView` with `editMode={false}`.** The public page and the editor preview share one component — a single source of truth, so page-to-page drift is structurally unlikely.
- **P1 · Drift risk lives *inside* `EditableProfileView` — block rendering forks into two internal branches:** the non-edit `switch (block.type)` at `EditableProfileView.tsx:154` and the editMode JSX at `:870`. Gallery/links render with different props per branch (editMode passes `onItemAdd`/`onDelete`, e.g. `:871` vs `:187`). · A new block type added to one branch but not the other renders correctly in only one mode. Consider a shared `renderBlockBody(block, theme)` consumed by both branches.
- **P1 · A *third* block-type switch exists for editor dispatch: `ProfileDashboard.tsx:399` (`case 'links'`/`case 'gallery'` → which editor to open).** Adding a block type means touching three switches (preview `:154`, editMode `:870`, dispatch `ProfileDashboard:399`). · Centralize the block-type → {block component, editor component} map in one registry.
- **P2 · Duplicated derivation: `(theme_json as any)?.avatar_url_page2` is computed in both `PublicProfile.tsx:254` and `EditableProfileView.tsx:1551`.** · Lift to a shared `getPage2Avatar(page)` helper so the two paths can't diverge.
